Comment lire cette page : chaque section décrit un module — ses entrées (données qu'il reçoit), son traitement (ce qu'il fait), ses sorties (ce qu'il produit). Les modules sont présentés dans l'ordre où ils sont appelés pendant une génération.

1. Chaîne de génération du site

Quand vous double-cliquez sur lancer.cmd, voici ce qui se passe dans l'ordre :

lancer.cmd
  → genere_site.py  (orchestre 5 phases)
        Phase 1 : documents.py   (scan + conversion PDF)
        Phase 2 : documents.py   (STRUCTURE.py par dossier)
        Phase 3 : builder.py     (HTML de navigation)
        Phase 4 : musique.py     (boutons YouTube sur PDF)
        Phase 5 : cree_table_des_matieres.py  (TDM globale)

2. Programmes principaux

lancer.cmd — v4.4 — Point d'entrée Windows

Entrées :

  • Argument optionnel nolocal : si présent, le serveur node.js ne démarre pas (utilisé par maj_github.py pour la génération sans consultation)
  • Environnement : détecte C: irpy13\Scriptsctivate.bat

Traitement :

  1. Passe en mode UTF-8 (chcp 65001) pour les accents dans les logs
  2. Calcule PROG_DIR (= dossier de lancer.cmd, c'est-à-dire prog\) et ROOT_DIR (= dossier parent, c'est-à-dire la racine du projet)
  3. Active l'environnement virtuel C: irpy13 (ou avertit si absent)
  4. Supprime entièrement html\ avant génération pour éviter les fichiers résiduels
  5. Exécute python genere_site.py depuis prog\
  6. Copie prog\style.css vers html\Hebreu4.0\html\style.css (nécessaire pour la consultation locale avec node.js)
  7. Lance npx http-server html -p 3500 (sauf si nolocal)

Sorties : dossier html\ reconstruit, serveur local sur le port 3500.

genere_site.py — v25.6 — Orchestrateur principal

Entrées :

  • settings.py : DOSSIER_DOCUMENTS, DOSSIER_HTML, BASE_PATH, CONFIG (configuration globale)
  • Le dossier documents\ et toute son arborescence (DOCX, PDF, CSV, HTML)
  • prog\style.css pour la copie vers html\

Traitement en 5 phases :

  • Init : crée le dossier html\, copie style.css, ouvre generation.log
  • Phase 1–2 : parcourt récursivement documents\ via os.walk(). Pour chaque dossier : appelle documents.generer_pdf_manquants() et documents.mettre_a_jour_structure()
  • Phase 3 : appelle builder.copier_fichiers_site()
  • Phase 4 : reparcourt documents\ pour les partitions (musique.traiter_partitions_du_dossier()), après la copie pour que les boutons écrasent les PDF sans bouton
  • Phase 5 : appelle cree_table_des_matieres.generer_tdm()

Sorties : dossier html\ complet avec pages HTML, PDF, CSS, et html dm\index.html. Fichier prog\generation.log.

settings.py — v1.1 — Configuration centrale

Entrées : aucune entrée dynamique — c'est un fichier de constantes édité manuellement.

Contenu et rôle de chaque variable :

  • DOSSIER_RACINE = C:\SiteGITHUB\Hebreu4.0 — racine absolue du projet (tous les chemins en découlent)
  • DOSSIER_DOCUMENTS = DOSSIER_RACINE\documents — dossier scanné par genere_site.py pour trouver les DOCX à convertir
  • DOSSIER_HTML = DOSSIER_RACINE\html — dossier de sortie du site généré
  • BASE_PATH = /Hebreu4.0/html — préfixe injecté dans tous les liens HTML (href, src). Vide ("") pour test local, /Hebreu4.0/html pour GitHub Pages
  • CONFIG["dossier_tdm"] = "tdm" — nom du sous-dossier TDM dans html\. En minuscules : GitHub Pages tourne sur Linux (sensible à la casse)
  • CONFIG["ignorer"] — liste des dossiers exclus du scan (nppBackup, .git, package...)
  • CONFIG["extensions_acceptees"] — extensions listées dans les index HTML (pdf, docx, html, jpg...)
  • CONFIG["regeneration"] — contrôle la conversion PDF : False=seulement si manquant, True=tout regénérer, "JJ/MM/AAAA"=regénérer si modifié après cette date
  • CONFIG["ajout_affichage"] = [" 📁 ", "", " 📘 ", ""] — préfixes/suffixes emojis ajoutés aux noms de dossiers et fichiers dans les index

Sorties : variables globales importées par tous les autres modules (from settings import DOSSIER_DOCUMENTS, DOSSIER_HTML, BASE_PATH, CONFIG).

documents.py — v2.2 — Gestion DOCX, PDF et STRUCTURE.py

Entrées (par dossier appelé depuis genere_site.py) :

  • Un dossier de documents\ avec ses fichiers DOCX, PDF, images
  • CONFIG["regeneration"] et CONFIG["regenerer_pdf_aujourd_hui"] pour la stratégie de reconversion

Traitement — generer_pdf_manquants(dossier) :

  • Liste tous les fichiers .docx du dossier (sauf préfixés __ qui sont des partitions traitées séparément)
  • Pour chaque DOCX, calcule le nom PDF normalisé via normaliser_nom() (accents supprimés, espaces → _, minuscules)
  • Compare la date du DOCX et du PDF existant selon la stratégie : absent, DOCX plus récent, date seuil, ou forcé
  • Si conversion nécessaire : appelle conversion_pdf.convertir_docx_vers_pdf()

Traitement — mettre_a_jour_structure(dossier) :

  • Lit le fichier STRUCTURE.py existant dans le dossier (ou crée un dict vide)
  • Ajoute les nouveaux fichiers détectés (PDF, HTML, images) avec leurs métadonnées : nom_html, nom_affiché, nom_TDM, position, affiché_TDM
  • Préserve la clé partitions (ajoutée par musique.py)
  • Sauvegarde STRUCTURE.py au format JSON dans le dossier

Sorties : fichiers PDF créés/mis à jour dans documents\, fichiers STRUCTURE.py mis à jour dans chaque sous-dossier.

builder.py — v1.2 — Génération HTML

Entrées :

  • Les fichiers STRUCTURE.py de chaque dossier (lus par lib/structure_utils.py)
  • Les fichiers optionnels entete.html, entete_general.html, pied.html, pied_general.html présents dans les dossiers sources
  • DOSSIER_TDM depuis settings.CONFIG pour construire le lien Sommaire

Traitement — generer_page_index(dossier) :

  • Construit la barre de navigation (fil d'Ariane + bouton "Sommaire" pointant sur BASE_PATH/tdm/index.html — variable, jamais hardcodé)
  • Charge l'en-tête HTML local ou général si présent
  • Génère le tableau de liens vers les fichiers listés dans STRUCTURE.py (avec préfixes emojis, noms affichés personnalisés)
  • Charge le pied de page HTML
  • Utilise BeautifulSoup pour indenter proprement le HTML

Traitement — copier_fichiers_site() :

  • Parcourt documents\ récursivement
  • Pour chaque dossier : crée le dossier miroir dans html\, génère son index.html, copie les fichiers aux extensions copiables (PDF, HTML, images, CSS, JS)

Sorties : arborescence html\ avec un index.html par dossier et tous les fichiers consultables.

musique.py — v1.14 — Partitions musicales

Entrées (par dossier) :

  • Fichier __correspondance.csv obligatoire dans le dossier : définit pour chaque partition le lien YouTube et les paramètres de boutons
  • Colonnes CSV principales : pdf_name, youtube_url, plus optionnellement : transparence, pos_x, pos_y, rotation, orientation, largeur, hauteur, ordre_boutons (0=YouTube|Jouer, 1=Jouer|YouTube)
  • Les PDF de partitions dans le dossier source

Traitement — traiter_partitions_du_dossier(dossier) :

  • Lit __correspondance.csv et récupère chaque ligne
  • Pour chaque partition : lit le PDF source, appelle place_bouton.py deux fois (bouton YouTube et bouton "Jouer ici" pointant sur le lecteur audio)
  • Le PDF résultant (avec boutons) est sauvegardé dans le dossier HTML mirror
  • En cas d'erreur, affiche un traceback complet pour diagnostic

Sorties : PDF modifiés avec boutons cliquables dans html\. La signature de traitement est enregistrée dans STRUCTURE.py.

cree_table_des_matieres.py — v6.33 — Table des matières globale

Entrées :

  • Tous les fichiers STRUCTURE.py de l'arborescence documents\ (lus récursivement)
  • Le dossier optionnel documents dm\ pour les templates HTML personnalisés (entete.html, pied.html, entete_general.html, pied_general.html)
  • CONFIG["dossier_tdm"] = "tdm" (nom du dossier de sortie)

Traitement — generer_tdm() :

  • Appelle construire_arbo_recursif() qui parcourt tous les dossiers et construit un arbre HTML de liens (<li>) vers chaque document, filtré par affiché_TDM: True dans STRUCTURE.py
  • Les noms affichés utilisent nom_TDM du STRUCTURE.py, avec résolution des templates {{nom_document_sans_ext}}
  • Applique les styles Markdown-like (**gras**, __italique__, [rouge]texte[/rouge]...)
  • Assemble le HTML final : en-tête + navigation + arbre + pied
  • Formatte avec BeautifulSoup (prettify())

Sortie : html dm\index.html — page de table des matières du site avec liens vers tous les documents marqués visibles.

3. Modules de conversion

conversion_pdf.py — v1.0 — Wrapper de conversion

Rôle : couche d'abstraction entre documents.py et docx_to_pdf.py. Isole la gestion des erreurs et l'import conditionnel de PDFCreator.

Entrées de convertir_docx_vers_pdf(docx_path, pdf_path, log_func) :

  • docx_path : chemin absolu du fichier DOCX source
  • pdf_path : chemin absolu du PDF à créer (peut être dans un autre dossier)
  • log_func : fonction de log injectée (affiche les messages dans la console et generation.log sans coupler conversion_pdf à genere_site)

Traitement : vérifie que PDFCreator est disponible, appelle docx_to_pdf.docx_to_pdf() avec timeout 60 secondes, capture FileNotFoundError, TimeoutError et exceptions génériques.

Sortie : True si PDF créé avec succès, False sinon.

docx_to_pdf.py — v1.3 — Conversion via PDFCreator

Prérequis système : Microsoft Word installé + PDFCreator installé et configuré en profil AutoSave vers C: emp\ + PyWin32.

Entrées de docx_to_pdf(docx_path, pdf_path, timeout=120) :

  • docx_path : chemin du DOCX à convertir
  • pdf_path : chemin de destination du PDF
  • timeout : délai max en secondes (défaut : 120s)

Traitement :

  1. Vérifie si Word est déjà ouvert (demande de le fermer si c'est le cas)
  2. Nettoie C: emp\ des PDF résiduels de précédentes conversions
  3. Ouvre le DOCX dans Word via COM (win32com.client)
  4. Imprime vers PDFCreator (imprimante virtuelle), qui dépose le PDF dans C: emp\
  5. Attend que le PDF apparaisse et soit déverrouillé (wait_for_file() scrute toutes les 0,5s)
  6. Déplace le PDF vers pdf_path via shutil.move()
  7. Supprime le résidu éventuel dans C: emp\

Sortie : fichier PDF créé à l'emplacement demandé. Chaque print() est flushé immédiatement (log en temps réel).

normalisation_utils.py — v1.0 — Règle unique de nommage

Problème résolu : plusieurs modules devaient normaliser des noms de fichiers pour les URLs (supprimer accents, gérer espaces). Chacun avait sa propre implémentation, ce qui créait des incohérences. Ce module est la source unique de vérité.

Fonctions exportées :

  • normaliser_nom(nom) — transformation principale : décomposition Unicode NFD → suppression des diacritiques (catégorie Mn) → apostrophes ' et ’ → _ → espaces → _ → minuscules. Exemples : 'Mi ha-ish.Ps 34.13-15.pdf' → 'mi_ha-ish.ps_34.13-15.pdf', "L'Auvergnat.pdf" → 'l_auvergnat.pdf'
  • nom_pdf_depuis_source(nom_fichier) — change l'extension en .pdf et normalise le stem. 'A A.docx' → 'a_a.pdf'
  • normaliser_chemin(chemin_relatif) — normalise chaque segment d'un Path séparément

Importé par : musique.py, documents.py, genere_site.py, builder.py.

place_bouton.py — v1.1 — Injection de bouton YouTube dans un PDF

Entrées de main(args) (appelé par musique.py) :

  • dossier_src, fichier_src : PDF source à modifier
  • dossier_dst, fichier_dst : PDF de sortie (peut être identique)
  • youtube_url : URL du lien cliquable
  • Paramètres optionnels : transparence (0-100), pos_x et pos_y en points PDF depuis le bas-gauche, rotation (N/E/S/O), orientation_texte (1-8), largeur, hauteur en points

Traitement :

  1. Crée un calque PDF transparent avec reportlab : dessine un rectangle rouge arrondi avec le texte "YouTube", appliqué avec la rotation et l'orientation demandées
  2. Fusionne ce calque sur la première page du PDF source avec pypdf (merge_page())
  3. Ajoute une annotation pypdf.annotations.Link avec la zone de clic correspondante et NewWindow=True pour ouvrir dans un nouvel onglet

Sortie : nouveau PDF avec bouton cliquable sur la première page. Le lien est une vraie annotation PDF (pas une image) : il fonctionne dans tous les lecteurs PDF.

4. Modules de déploiement et maintenance

remplace.py — v21 — Déploiement package → prog + manuels

Entrées :

  • Dossier package\prog\ : fichiers sources versionnés (nom_vX.Y.ext)
  • Dossier package\manuels\ : manuels HTML versionnés (nom_vX.Y.html)

Traitement :

  1. Auto-découverte : scan récursif de package\prog\ (sous-dossiers inclus sauf archive\ et ancien\). Pour chaque fichier avec suffixe _vX.Y : déduit le nom cible (sans version) et le dossier de destination
  2. Sélection version MAX : si maj_github_v1.9.py et maj_github_v1.17.py coexistent, seul v1.17 est déployé. Comparaison numérique (1.17 > 1.9) et non alphabétique
  3. Décision de copie : copie si version source ≠ version cible, ou si la date du fichier source est plus récente de plus de 2 secondes
  4. Suppressions : efface les anciens fichiers obsolètes (lib1\, anciens shims, docx déplacés dans manuels\)
  5. Rapport "non répertoriés" : liste les fichiers présents dans prog\ qui ne correspondent à aucun fichier source connu
  6. Style.css local : copie prog\style.css vers html\Hebreu4.0\html\ pour la consultation locale avec node.js

Sorties : prog\ et manuels\ synchronisés avec les dernières versions de package\.

maj_github.py — v1.17 — Publication sur GitHub

Entrées :

  • prog\config.yaml : racine_source, racine_site_local, lancer_cmd, url_github, branche, github_token (optionnel), github_user (optionnel), message_commit, confirmation_git

Traitement en 4 étapes (interface graphique Tkinter) :

  1. Étape 1 — Sync sources : appelle sync_dossiers.py pour copier les fichiers modifiés de racine_source vers racine_site_local
  2. Étape 2 — Génération : lance lancer.cmd nolocal via subprocess.Popen en streaming (lecture ligne par ligne pour affichage en temps réel sans blocage)
  3. Étape 3 — Commit : git add -A + git commit avec message automatique ou fixé
  4. Étape 4 — Push :
    • Si github_token renseigné : vérifie le token via l'API api.github.com/user (affiche le login et les 4 premiers/derniers caractères du token pour diagnostic), puis injecte Authorization: Basic base64(user:token) via -c http.extraheader
    • Si token absent : utilise GCM (Git Credential Manager de GitHub Desktop). Nécessite que git-credential-manager.exe soit autorisé dans le pare-feu Windows
    • Lance git push origin main via subprocess.Popen (streaming, GCM peut interagir avec Windows si besoin)

Sorties : site publié sur GitHub Pages. Le token n'apparaît jamais dans les logs (masqué après usage).

sync_dossiers.py — v1.1 — Synchronisation dossiers

Entrées :

  • config.yaml (chemin passé en argument ou indiqué par drag&drop) avec : source, destination, recent (true/false), filtre (regex séparées par virgules), nouveau

Traitement :

  1. Parcourt récursivement source et compare avec destination
  2. Sélectionne les fichiers selon les critères : date plus récente si recent=true, absent si nouveau=true, correspondant au filtre regex
  3. Affiche une fenêtre Tkinter avec la liste des fichiers à copier (cases à cocher pour en exclure)
  4. Copie les fichiers cochés avec shutil.copy2() (préserve les dates)

Sortie : fichiers syncés. Message uniquement en cas d'erreur (mode silencieux si tout se passe bien).

installer.py — v1.1 — Création de l'environnement Python

Entrées : aucune — exécuté directement par l'utilisateur. Peut être relancé sans risque si l'environnement existe déjà.

Traitement :

  1. Vérifie Python ≥ 3.11 (bloquant si inférieur)
  2. Crée C: irpy13\ via python -m venv (ou vérifie que Scripts\pip.exe existe si le dossier est déjà présent)
  3. Met à jour pip dans le venv
  4. Pour chaque package : vérifie avec pip show s'il est déjà installé, sinon exécute pip install
  5. Vérifie les imports dans le venv via subprocess (test isolé)

Packages installés : pyyaml, python-docx, pypdf, reportlab, psutil, beautifulsoup4 (requis pour la TDM), requests.

Sortie : C: irpy13\ prêt à l'emploi, activé par lancer.cmd et MAJ_GITHUB.cmd.

versions.py — v1.8 — Vérification de l'installation

Entrées : aucune — scanne prog\ et prog\lib\ depuis son propre emplacement.

Traitement :

  1. Scan dynamique : liste tous les .py de prog\ et prog\lib\ via glob(). Pour chaque fichier : extrait la version via ast.parse() (analyse l'AST, pas d'import) et compare à la version minimale attendue
  2. Autres fichiers : vérifie présence de lancer.cmd, config.yaml, style.css, remplace.py
  3. Config.yaml : parse avec le parser clé = valeur (PyYAML échoue sur ce format non standard), vérifie la présence des 7 champs obligatoires, détecte les valeurs modèle non remplacées
  4. Token GitHub : si token et user renseignés, appelle api.github.com/user avec l'en-tête Authorization pour valider le token et vérifier la correspondance du login

Sortie : rapport coloré en console (ANSI) avec bilan OK/ATTENTION/ERREURS. Code de retour 0 si tout est OK, 1 sinon (utilisable dans des scripts batch).

5. Modules de bibliothèque (prog\lib\)

Ces modules ne sont jamais appelés directement — ils sont importés par les programmes principaux.

lib\structure_utils.py — v2.1 — Gestion de STRUCTURE.py

Rôle : chaque dossier de documents\ possède un fichier STRUCTURE.py (format JSON malgré l'extension .py) qui mémorise les métadonnées de chaque document pour personnaliser leur affichage.

Fonctions principales :

  • charger_structure(dossier) : lit STRUCTURE.py et retourne un dict {"dossiers": [...], "fichiers": [...]}
  • sauvegarder_structure(dossier, struct) : écrit le dict en JSON. Fix v2.1 : json.dumps écrit null, converti en None Python lors de la relecture
  • resoudre_templates_runtime(item, variables) : résout les templates {{nom_document_sans_ext}}, {{titre_dossier}} dans les champs nom_affiché, nom_TDM, etc. Plusieurs passes pour les templates imbriqués (max 5)

lib\partition_utils.py — v2.3 — Boutons PDF partitions

Rôle : version avancée de place_bouton.py pour les partitions musicales avec deux boutons et paramètres CSV complets.

Paramètres CSV gérés : position X/Y en pourcentage de la page, transparence, rotation, orientation, ordre des boutons (YouTube avant ou après "Jouer ici"), position X négative depuis le bord droit.

Toutes les opérations PDF utilisent le système de coordonnées CTM (Current Transformation Matrix) de PDF, avec conversion vers MediaBox pour les zones de clic — compatible avec les PDF en rotation (pages en paysage).

lib\html_utils.py — v1.0 — Templates HTML

Chargement et interprétation des fichiers templates HTML (entete.html, pied.html...). Remplace les variables {{BASE_PATH}} et gère les commentaires de débug de structure.

lib ichier_utils.py — v1.1 — Filtrage de fichiers

Détection des fichiers spéciaux (préfixés __ comme __correspondance.csv, __partition_*.docx), filtrage par extension, détection des fichiers entete/pied à exclure des listes de documents.

lib\pdf_utils.py — v1.1 — Logique de régénération PDF

Fonctions utilitaires pour décider si un PDF doit être reconverti : comparaison de dates DOCX/PDF, détection "créé aujourd'hui". Fix v1.1 : évite la reconversion en boucle lorsque le PDF est créé dans la même journée.
