Versioning, organisation et bonnes pratiques

1. Principes fondamentaux

Règle d'or
Un fichier source modifié = numéro de version incrémenté.
Le dossier package\ est l'unique source de vérité historique.
Le dossier prog\ ne contient que les versions actives déployées.
Aucun fichier sensible (token, mot de passe) ne doit être commité sur git.

2. Numérotation des versions

2.1 Schéma général

FormatSignificationExemple
X.YX = version majeure, Y = correction/amélioration mineuregenere_site_v25.3.py
X_YVariante pour les fichiers .cmd (le point est interdit dans les noms batch)lancer_v4_2.cmd

2.2 Quand incrémenter

ChangementVersion
Correction de bug, ajout mineurY+1 (ex : 1.2 → 1.3)
Refonte fonctionnelle, nouveau moduleX+1, Y=0 (ex : 1.7 → 2.0)
remplace.py : à chaque livraisonY+1 même si un seul autre fichier change

3. Structure obligatoire d'un fichier Python

# nom_fichier_vX.Y.py — Version X.Y
# Description courte du module
# v X.Y : description du changement
# v X.Y-1 : ...

version = ("nom_fichier.py", "X.Y")
print(f"[Import] {version[0]} - Version {version[1]} chargé")

# ... code ...

# fin de nom_fichier_vX.Y.py - Version X.Y

4. Organisation des dossiers

DossierContenuRègle
package\prog\Sources versionnéesNoms avec suffixe _vX.Y — remplace.py garde la version MAX
package\prog\archive\Anciennes versions à conserverIgnoré par remplace.py
prog\Fichiers déployés actifsNoms sans version, copiés par remplace.py
manuel\DocumentationUn sous-dossier par composant + _general\

5. Convention lib_ (modules de bibliothèque)

Les modules destinés à prog\lib\ sont préfixés lib_ dans package\prog\ :

SourceDéployé vers
lib_html_utils_v1.0.pyprog\lib\html_utils.py
lib_structure_utils_v2.1.pyprog\lib\structure_utils.py

Alternative : créer un sous-dossier package\prog\lib\ — il sera mirroisé vers prog\lib\.
